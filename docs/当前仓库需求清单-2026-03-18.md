# 当前仓库需求清单

> 基于当前仓库代码、README、CHANGELOG、现有页面/Hook/Store/Tauri-Rust 服务整理。目标是形成一份“简洁但不漏项”的现状需求清单，便于后续做需求拆解、重构或补文档。

## 1. 产品范围

- 产品是一个仅面向 Linux 的玲珑应用商店社区版桌面客户端。
- 产品必须基于 `React + TypeScript + Vite + Ant Design + Tauri 2 + Rust` 运行。
- 产品目标是管理玲珑应用生命周期，而不是重新实现玲珑包管理器。
- 产品同时依赖本地玲珑能力与远程商店接口，前者负责真实系统操作，后者负责元数据、推荐、分类、更新校验、反馈和统计。

## 2. 用户功能需求

### 2.1 启动与环境门禁

- 应用启动时必须先展示启动页，并给出初始化进度和当前步骤。
- 应用进入主界面前必须检测玲珑环境、架构、仓库、系统版本、glibc、内核信息和 `ll-cli` 版本。
- 当系统未安装玲珑环境或环境不可用时，必须阻止进入主界面。
- 环境异常时必须提供退出商店、手动安装、自动安装、重新检测四种处理路径。
- 自动安装玲珑环境后，应用应支持直接重新检测并继续进入，无需手动重启。

### 2.2 全局框架与导航

- 应用必须提供自定义标题栏，支持最小化、最大化/还原、关闭。
- 标题栏必须提供全局搜索入口，并支持回车搜索与一键清空。
- 应用必须提供侧边栏主导航，至少覆盖推荐、分类、更新、我的应用、设置和下载管理入口。
- 侧边栏必须支持服务端下发的自定义分类菜单。
- 菜单必须支持更新数量等角标提醒。
- 应用必须提供下载管理弹窗，支持随时查看安装队列与进度。

### 2.3 应用发现与浏览

- 应用必须提供推荐页，包含轮播推荐和推荐应用列表。
- 应用必须提供分类页，支持分类切换、分类栏展开/折叠、分页加载和无限滚动。
- 应用必须提供搜索结果页，支持关键字搜索、防抖和分页加载。
- 应用必须提供排行榜能力，至少包含最新上架榜和下载量榜。
- 应用必须提供自定义专题页，支持按后台菜单加载列表。
- 自定义专题页必须支持排序和过滤低分应用。
- 列表页必须明确区分首屏加载、分页加载、空状态和“没有更多数据”状态。

### 2.4 应用详情与版本操作

- 应用必须提供详情页，展示基础信息、截图、版本列表和安装状态。
- 详情页必须支持复制应用 ID。
- 详情页必须支持查看并操作不同版本。
- 详情页必须支持安装指定版本、升级、打开、卸载指定版本。
- 当安装低版本覆盖高版本时，必须给出降级确认。
- 当安装已存在版本时，必须支持强制重装确认。
- 当卸载运行中应用时，必须给出强制关闭并卸载的确认。
- 已安装应用在详情页必须支持创建桌面快捷方式。

### 2.5 安装、更新、卸载与运行

- 应用必须支持安装玲珑应用，并通过队列串行执行。
- 应用必须支持运行已安装应用。
- 应用必须支持卸载指定应用版本。
- 应用必须支持检测已安装应用更新。
- 应用必须提供更新页，展示全部可更新应用，并支持手动检查更新。
- 更新页必须支持一键将全部待更新应用加入安装队列。
- 安装、卸载、更新完成后，必须同步刷新已安装列表、更新列表和菜单红点。

### 2.6 下载管理与安装进度

- 应用必须展示当前安装任务、等待队列和最近安装历史。
- 下载管理必须展示每个任务的状态、进度、错误信息和操作入口。
- 用户必须能够移除待安装任务。
- 用户必须能够取消当前正在执行的安装任务。
- 安装成功后必须允许直接打开应用。
- 下载管理必须支持清理历史记录。
- 下载管理必须显示实时网络速度。
- 应用重启后必须能够识别并恢复中断安装任务的状态。

### 2.7 我的应用与进程管理

- 应用必须提供“我的应用”视图，展示已安装应用。
- 已安装列表必须支持按 `appId` 合并多版本，并展示最高版本代表项。
- 应用必须提供“玲珑进程”视图，展示当前运行中的玲珑应用。
- 进程页必须支持手动刷新、自动刷新、错误提示和最后刷新时间展示。
- 进程页必须支持停止进程。
- 进程页必须支持复制进入容器命令、应用 ID、PID、容器 ID。

### 2.8 设置、关于与反馈

- 设置页必须支持“启动时检查商店版本更新”开关。
- 设置页必须支持“容器内自动更新商店本体”开关。
- 设置页必须支持“是否显示基础运行服务”开关。
- 设置页必须支持清理废弃基础服务。
- 关于页必须展示商店版本、玲珑组件版本、官网/仓库地址和已收录应用数量。
- 关于页必须支持检查商店新版本。
- 关于页必须支持意见反馈。
- 反馈功能必须支持可选上传本地日志文件。

## 3. 系统与后端能力需求

- 前端必须通过 Tauri IPC 调用 Rust 命令，不直接在页面中散落 shell 调用。
- Rust 后端必须负责调用 `ll-cli` 执行安装、卸载、运行、列出应用、搜索远程应用、查看内容、查询进程、停止进程、清理基础服务等系统操作。
- Rust 后端必须对 `ll-cli` 输出做结构化解析，并返回前端可直接消费的数据。
- 本地已安装应用数据必须与远程商店详情数据合并，以补全图标、中文名、分类和描述。
- 安装流程必须通过事件把进度、状态、错误码和错误详情持续推送给前端。
- 商店自更新必须通过独立事件通知前端“已可重启生效”。
- 容器环境下必须支持后台静默检查并替换商店二进制。
- 网络测速必须由本地能力提供，不能依赖前端估算。
- 创建桌面快捷方式必须由后端完成，目标是用户桌面目录。

## 4. 数据与集成需求

- 远程接口必须覆盖分类、列表、详情、轮播、推荐、自定义菜单、应用更新校验、反馈、日志上传、匿名统计和环境安装脚本。
- 应用更新校验必须基于“本地已安装版本 + 远端版本信息”批量计算。
- 商店自身版本更新必须基于 Gitee Release 信息判断。
- 应用必须记录匿名启动访问数据和安装/卸载统计数据。

## 5. 非功能需求

- 应用仅支持 Linux，重点兼容 `x86_64` 与 `arm64/aarch64`。
- 启动阶段应优先完成环境门禁和主界面可用性，非关键任务应异步执行。
- 应用列表必须支持分页、无限滚动、骨架屏和缓存回填，避免首屏长时间白屏。
- 列表缓存必须采用“内置 seed + 运行时本地缓存 + 页面可见时后台刷新”的混合策略。
- KeepAlive 页面在隐藏时必须停止滚动监听、自动补页、尺寸观察和轮询等持续副作用。
- 所有关键 `ll-cli` 调用都必须具备超时控制、失败分类和日志记录。
- 解析 `ll-cli` 输出时必须尽量固定英文 locale，降低文本解析不确定性。
- 界面必须对加载、空态、错误态、成功态和可预期失败给出明确反馈。
- 安装、卸载、更新、进程停止等高风险操作必须有明确确认或保护机制。

## 6. 工程约束需求

- 跨页面复用的业务逻辑必须统一收口到共享 Hook/Store，避免在页面中重复实现副作用。
- 安装、卸载、红点、列表缓存、页面可见态、更新检测等行为必须走统一入口。
- 优先复用现有 Rust 命令与 IPC 事件，不新增不可控的前端 Shell 调用。
- 新增或变更能力时，前后端类型、运行时校验、调用封装和变更记录必须保持同步。
