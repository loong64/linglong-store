# 代码逻辑冗余整改计划（2026-03-10）

## 文档目的

本计划基于 `docs/代码逻辑冗余审查报告-2026-03-10.md`，将“问题清单”进一步转化为**可执行、可排期、可验收**的整改路线图。

本计划遵循以下原则：

- **先收敛事实来源，再收敛动作入口，最后收敛基础设施**
- **优先低风险高收益改动，避免一口气全仓大手术**
- **保持现有业务行为稳定，不在同一阶段同时改太多方向**
- **优先复用现有 hook/store/service，不引入新的分散入口**
- **所有改造以“统一入口、单一职责、边界清晰”为目标**

> 说明：本文件是整改方案，不包含代码修改。本阶段仍属于“方案确认”范围。

## 整改总览

### 总体目标

通过 4 个阶段完成冗余逻辑治理：

1. **阶段一：统一事实来源**
2. **阶段二：统一前端动作编排与页面状态机**
3. **阶段三：治理前后端契约与数据边界**
4. **阶段四：升级 Rust `ll-cli` 适配层**

### 预期收益

完成后应明显改善：

- 启动链路稳定性
- store / hook / page 之间的职责边界
- 页面层重复逻辑数量
- IPC 契约一致性
- Rust 服务层的可维护性与可扩展性

### 执行策略

建议采用 **小步快跑 + 多 PR 分批落地**：

- 每个阶段拆成 2~4 个小任务
- 每个任务应尽量控制在单一主题
- 每个任务完成后都要进行局部验证
- 每个阶段结束后做一次回归审查

## 阶段一：统一事实来源

### 阶段目标

先解决“一个事实有多个主人”的问题，因为这类问题最容易造成状态错乱、重复请求和后续逻辑扩散。

### 任务 1.1：统一启动初始化入口

**目标**

让“加载已安装应用、补全详情、检查更新、恢复安装任务”拥有单一启动编排入口。

**涉及文件**

- `src/hooks/launch.ts`
- `src/layout/index.tsx`
- `src/stores/installedApps.ts`
- `src/stores/updates.ts`

**现状问题**

- `useLaunch.initialize()` 与 `AppLayout` 都在触发初始化相关逻辑
- `fetchInstalledApps()` 与 `updateAppDetails()` 存在链路重叠
- `recoverInstallTask(installedApps)` 存在读取旧快照风险

**整改动作**

1. 明确 `useLaunch` 是否作为唯一启动编排入口
2. 收掉 `AppLayout` 中重复的初始化型触发逻辑
3. 让 `fetchInstalledApps()` 与 `updateAppDetails()` 的职责边界变清晰：
   - 要么 `fetchInstalledApps()` 完整负责“列表 + 详情”
   - 要么两者拆清并只由一个上层编排调用
4. 修复 `recoverInstallTask()` 使用旧 `installedApps` 快照的问题

**完成标准**

- 启动期每类核心请求只保留一个入口
- 不再出现重复调用 `updateAppDetails()`
- 安装任务恢复使用的是最新状态数据

**风险等级**

- 中

**建议 PR 名称**

- `refactor(launch): unify bootstrap flow ownership`

---

### 任务 1.2：统一环境状态单一来源

**目标**

让环境检测状态只保留一套业务真相，避免 `global store` 与 `useLaunch` 双轨制。

**涉及文件**

- `src/hooks/useLinglongEnv.ts`
- `src/hooks/launch.ts`
- `src/layout/launchPage/index.tsx`
- `src/components/LinglongEnvDialog/index.tsx`
- `src/stores/global.ts`

**整改动作**

1. 确认环境状态的 canonical source：
   - 推荐：环境业务状态放 store，`useLaunch` 只做流程编排
2. 清理重复的 `envReady / envChecked / error` 表达
3. 统一重试流程：避免 `LinglongEnvDialog` 与 `retry()` 同时重复触发环境检测
4. 评估 `checked` 等遗留字段是否继续保留

**完成标准**

- `envReady` / `checking` / `reason` 只有一套业务真相
- 重试一次只执行一轮必要检测
- 启动页与环境弹窗不再“各自推进流程”

**风险等级**

- 中

**建议 PR 名称**

- `refactor(env): consolidate env state ownership`

## 阶段二：统一前端动作编排与页面状态机

### 阶段目标

减少页面层“重复实现同一套业务逻辑”的情况，把公共动作和公共状态机收敛到 hook / selector / 连接层。

**目标**

替换 5 个列表页重复的分页状态逻辑。

**涉及文件**

- `src/pages/allApps/index.tsx`
- `src/pages/recommend/index.tsx`
- `src/pages/ranking/index.tsx`
- `src/pages/searchList/index.tsx`
- `src/pages/customCategory/index.tsx`
- `src/hooks/useAutoLoadWhenNotScrollable.ts`

**建议方案**

抽出一个通用 hook，例如：

- `usePaginatedAppList()`
- 或按查询类型拆成更细的分页 domain hook

**建议承载能力**

- `initialLoading`
- `loading`
- `hasMore`
- `pageNo`
- `items`
- 参数变化后的 reset
- `loadNextPage()`
- 统一的追加合并策略

**完成标准**

- 5 个列表页不再各写一套分页状态机
- 页面仅保留查询参数、场景 UI、空态差异
- `useAutoLoadWhenNotScrollable` 专注“什么时候触发加载”，分页 hook 专注“如何维护分页状态”

**风险等级**

- 中

**建议 PR 名称**

## 阶段三：治理前后端契约与数据边界

### 阶段目标

让 IPC DTO、前端增强模型、视图模型不再混在一起，并逐步补齐运行时校验。

### 任务 3.1：统一 Tauri client 入口

**目标**

消除组件层和历史 service 对 `invoke()` 的直连，统一走单一 client。

**涉及文件**

- `src/apis/invoke/index.ts`
- `src/services/installService.ts`
- `src/components/speedTool/index.tsx`
- 未来建议新增：`src/services/tauri.ts`

**整改动作**

1. 定义唯一 Tauri client 封装位置
2. 将 `speedTool` 改为复用统一 client
3. 评估 `installService.ts` 是清理、降级为 helper，还是并入主入口
4. 为统一 client 预留 timeout / error transform / validation 接口

**完成标准**

- 组件层不再直接调用 `invoke`
- `install_app` 只保留一个前端调用入口
- 小型命令（如网速）也进入统一契约管理

**风险等级**

- 低到中

**建议 PR 名称**

- `refactor(ipc): consolidate tauri client entry`

### 任务 3.2：拆分 IPC DTO 与前端增强模型

**目标**

解决 `InstalledApp` 一类模型同时扮演传输类型、store 模型、页面模型的混用问题。

**涉及文件**

- `src/types/api/invoke.d.ts`
- `src/stores/installedApps.ts`
- `src/stores/updates.ts`
- `src/pages/appDetail/index.tsx`
- `src/pages/myApps/components/myApp/index.tsx`
- Rust 对应 model 文件

**建议方案**

至少拆成三层：

1. `TauriInstalledApp`（严格对应 Rust）
2. `EnrichedInstalledApp`（补本地/远端详情）
3. 页面专用 view model（如需要）

**整改动作**

1. 给现有类型重新命名，避免继续混用
2. 梳理哪些字段来自 Rust，哪些字段来自前端 enrich
3. 把页面临时字段从基础 DTO 中剥离出去

**完成标准**

- DTO 只表达传输边界
- enrich 逻辑只在明确层次发生
- 页面不再把 UI 字段回写到 IPC DTO 类型里

**风险等级**

- 中

**建议 PR 名称**

- `refactor(types): split ipc dto from enriched models`

## 阶段四：升级 Rust `ll-cli` 适配层

### 阶段目标

把后端从“多个 service 各自包命令、解析输出、拼错误字符串”升级成“统一 `ll-cli` adapter 基础设施”。

### 任务 4.1：建立统一 `ll-cli executor`

**目标**

统一命令执行、输出提取、超时策略和错误表达。

**涉及文件**

- `src-tauri/src/services/mod.rs`
- `src-tauri/src/services/linglong.rs`
- `src-tauri/src/services/process.rs`
- `src-tauri/src/services/prune.rs`
- `src-tauri/src/services/install/operations.rs`
- `src-tauri/src/services/linglong_env.rs`

**建议方案**

建立 `ll-cli executor`，统一承载：

- locale 注入
- 命令执行
- timeout
- stdout/stderr 提取
- 非零退出码处理
- 错误类型转换

**完成标准**

- 主要 `ll-cli` 命令不再各自直接 `.output()`
- 不同 service 对命令失败的处理方式趋于一致

**风险等级**

- 中到高

**建议 PR 名称**

- `refactor(rust): introduce ll-cli executor`

---

### 任务 4.2：建立公共 parser / models

**目标**

让 `list/search/repo/version/ps` 相关输出解析不再散在各 service 中。

**涉及文件**

- `src-tauri/src/services/install/json_parser.rs`
- `src-tauri/src/services/install/models.rs`
- `src-tauri/src/services/process.rs`
- `src-tauri/src/services/linglong.rs`
- `src-tauri/src/services/linglong_env.rs`

**整改动作**

1. 抽公共 `models`
2. 抽公共 `parser`
3. 让 `process.rs` 不再定义与 `install/models.rs` 重复的结构
4. 保持 parser 只做文本到结构的转换，不掺业务决策

**完成标准**

- 同一类 `ll-cli list/search` 输出只有一套解析实现
- 共享模型不再在多个 service 中重复定义

**风险等级**

- 中

**建议 PR 名称**

- `refactor(rust): extract shared ll-cli parsers and models`

---

### 任务 4.3：统一 timeout / blocking 策略

**目标**

解决“表面 async，内部 blocking”的结构性问题。

**涉及文件**

- `src-tauri/src/services/**/*.rs`
- 尤其是：
  - `install/installer.rs`
  - `linglong_env.rs`
  - `process.rs`
  - `operations.rs`

**整改动作**

1. 对命令执行统一设置 timeout 策略
2. 评估使用 `tokio::process::Command` 或 `spawn_blocking`
3. 替换 `std::thread::sleep(...)` 为更符合异步上下文的方式
4. 补齐 `stderr` drain 处理，避免潜在阻塞

**完成标准**

- 主要命令执行不再直接阻塞 async 上下文
- timeout 表达统一
- 长任务输出处理更加健壮

**风险等级**

- 高

**建议 PR 名称**

- `refactor(rust): normalize command timeout and async execution model`

---

### 任务 4.4：拆分职责过宽的 Rust 服务文件

**目标**

降低 `linglong_env.rs`、`install/operations.rs` 的职责堆积。

**涉及文件**

**完成标准**

- 大文件职责清晰
- 死代码 suppress 缩小到必要范围
- 模块职责更可维护

**风险等级**

- 中

**建议 PR 名称**

- `refactor(rust): split oversized linglong service modules`

## 验收建议

### 阶段一验收

- 启动期关键请求次数明显减少
- 环境重试链路不再重复检测
- 菜单红点只由单一 selector 驱动
- 安装取消事件语义正确

### 阶段二验收

- 主要列表页分页逻辑显著收敛
- 不再存在 `empty-*` 假数据拼接
- `ApplicationCard` 主要页面接入统一连接层
- 安装 / 卸载 / 批量更新的后置同步逻辑统一

### 阶段三验收

- 组件层不再直接 `invoke`
- `InstalledApp` 等基础类型边界清晰
- 高频 IPC 有 runtime validation
- HTTP / IPC 错误表达统一性提升

### 阶段四验收

- 主要 Rust service 不再各自重复 `.output() + parse + stderr`
- `ll-cli` 基础设施形成统一 adapter
- 大型 Rust 服务文件职责更清楚

## 不建议的做法

在整改过程中，建议避免以下做法：

1. **不要一边改事实来源，一边改 UI 展示结构**
   - 容易把状态治理和视觉变更搅在一起，增加回归难度。

2. **不要在阶段一就试图完成完整 DTO 分层**
   - 会把低风险阶段变成大范围类型迁移，收益反而下降。

3. **不要把 Rust executor、parser、timeout 治理打成一个超大 PR**
   - 这类基础设施改动最怕一次性全改，建议拆小步。

4. **不要再新增页面级直连 `invoke` 或页面级自建 selector**
   - 否则一边治理，一边继续制造新的冗余。

## 结语

这份整改计划的核心思路可以概括成一句话：

> **先让“谁说了算”变清楚，再让“谁来做”变统一，最后再把底层基础设施升级到可复用。**

如果按这个顺序推进，改造成本会更可控，回归风险也更低，不至于把本来是“治技术债”的事做成“再造一次技术债”。
