# 代码逻辑冗余审查报告（2026-03-10）

## 审查背景

本次审查目标是对仓库进行**分模块并行逻辑审查**，重点识别以下问题：

- 重复逻辑与重复副作用
- 可以收敛到统一入口的业务流程
- 状态职责重叠、派生逻辑分散
- 前后端契约边界不清
- Rust `ll-cli` 适配层重复实现
- 潜在死代码、薄封装、遗留抽象

本次审查为**只读审查**，**未修改任何业务代码**。

## 审查方式

本次通过多路并行审查完成，覆盖以下范围：

- `src/hooks`
- `src/components`
- `src/pages`
- `src/apis` / `src/services` / `src/util` / `src/constants`
- `src/stores` / `src/layout` / `src/router`
- `src/apis/invoke` / `src/types` / `src-tauri/src/lib.rs`
- `src-tauri/src/services` / `src-tauri/src/utils`

并对部分高优先级结论进行了二次核对，确认存在真实代码证据，而不是仅基于命名推断。

## 总体结论

当前代码库的问题不是“某一个文件写坏了”，而是已经出现了比较典型的 **职责分散** 与 **逻辑重复**：

1. **同一类状态被多个层同时拥有**：页面、hook、store、组件各自维护一部分。
2. **同一类动作流没有统一入口**：启动初始化、环境检测、安装/卸载后的刷新链路分布在多个模块。
3. **数据边界没有真正收口**：IPC、HTTP、前端 enrich/view model 混住，运行时校验不足。
4. **Rust 后端已有 `ll-cli` 命令工厂，但没有真正的执行器 / 解析器 / 统一错误模型**。

从维护成本角度看，仓库当前已经处于：

- **交互入口开始统一**，例如卸载流程、菜单红点、自动补页已有收敛意识；
- **副作用编排和派生状态仍未彻底统一**，这正是冗余逻辑的主要来源。

## 优先级最高的 12 个问题

### P0-1 启动初始化链路重复执行

**涉及模块**：`src/hooks/launch.ts`、`src/layout/index.tsx`、`src/stores/installedApps.ts`、`src/stores/updates.ts`

**问题描述**：

启动阶段“加载已安装应用、补全详情、检查更新”被多个入口同时触发：

- `useLaunch.initialize()` 会执行已安装应用拉取与更新检查
- `AppLayout` 挂载后也会触发 `fetchInstalledApps(showBaseService)`
- `installedAppsStore.fetchInstalledApps()` 内部已包含详情补全
- `useLaunch.initialize()` 后续又再次调用 `updateAppDetails()`

**已核对证据**：

- `src/hooks/launch.ts` 同时解构了 `fetchInstalledApps`、`updateAppDetails`、`installedApps`
- 同文件中存在 `await fetchInstalledApps(showBaseService)`、`await updateAppDetails()`、`recoverInstallTask(installedApps)`
- 并且单独读取了 `checkUpdates`

**影响**：

- 启动期重复请求
- 状态覆盖顺序不清
- 后续很难判断“到底谁拥有初始化职责”

**建议方向**：

收敛为单一启动编排入口，例如由 `useLaunch` 独占启动编排，`AppLayout` 只消费结果。

---

### P0-2 启动链路读取了可能过期的 `installedApps` 快照

**涉及模块**：`src/hooks/launch.ts`

**问题描述**：

`useLaunch` 在同一次异步初始化链路里：

- 先调用 `fetchInstalledApps(showBaseService)`
- 后续调用 `recoverInstallTask(installedApps)`

但 `installedApps` 是当前 render 时解构出来的 store 快照，不保证是 `fetchInstalledApps` 执行后的最新值。

**影响**：

- 中断安装恢复可能使用旧数据
- 可能误判任务是否已经安装成功

**建议方向**：

- 要么让 `fetchInstalledApps()` 返回最新数据
- 要么恢复前直接通过 `useInstalledAppsStore.getState()` 读取最新值

---

### P0-3 环境检测状态双轨制：全局 store 一套，`useLaunch` 本地一套

**涉及模块**：`src/hooks/useLinglongEnv.ts`、`src/hooks/launch.ts`、`src/layout/launchPage/index.tsx`、`src/stores/global.ts`

**问题描述**：

“环境是否可用 / 是否已检测”同时存在两套状态源：

- 全局 store：`checking`、`checked`、`envReady`、`reason`
- `useLaunch` 本地：`envReady`、`envChecked`、`error`

`LinglongEnvDialog` 与 `useLaunch.retry()` 还会导致一次用户重试流程里环境检查重复执行。

**影响**：

- 状态归属不明确
- UI 可能消费不同事实来源
- 重试、自动安装、初始化之间时序复杂

**建议方向**：

明确唯一事实来源：

- 要么环境状态统一归 global store
- 要么 `useLaunch` 管流程，store 只存必须全局共享的信息

---

### P0-4 安装 / 卸载后的后置同步链路没有统一入口

**涉及模块**：`src/hooks/useGlobalInstallProgress.ts`、`src/hooks/useAppUninstall.ts`、`src/stores/installedApps.ts`、`src/stores/updates.ts`

**问题描述**：

安装成功和卸载成功本质上都属于“应用集合发生变化后的同步收尾”，但现在分散在不同 hook：

- 安装成功：刷新列表、刷新更新、埋点、提示
- 卸载成功：删除本地项、刷新更新、埋点、提示

同时 `checkUpdates()` 内部还会再次拉取已安装应用，形成重复刷新。

**影响**：

- 同一类后置动作多处分散
- 刷新策略不一致
- 容易出现“每个流程都各自补一点”的继续扩散

**建议方向**：

抽统一后置同步入口，例如：

- `syncAfterAppSetChanged()`
- 内部统一负责 revalidate、乐观更新、埋点、提示

### P0-5 列表页分页状态机重复出现在 5 个页面

**涉及模块**：

- `src/pages/allApps/index.tsx`
- `src/pages/recommend/index.tsx`
- `src/pages/ranking/index.tsx`
- `src/pages/searchList/index.tsx`
- `src/pages/customCategory/index.tsx`

**问题描述**：

这些页面都在重复维护类似状态：

- `pageNo`
- `loading`
- `initialLoading`
- `totalPages`
- `loadNextPage()`
- 参数变化后的 reset
- 追加拼接逻辑

虽然已有 `useAutoLoadWhenNotScrollable` 统一滚动触发，但分页状态机本身并未收口。

**影响**：

- 页面间重复逻辑很多
- 维护成本高
- 稍改一个列表行为，多个页面都要同步改

**建议方向**：

抽一个统一分页列表 hook，让页面只保留查询参数和 UI 差异。

### P0-6 列表占位策略存在历史残留，`empty-*` 清理逻辑扩散到多页

**涉及模块**：

- `src/pages/recommend/index.tsx`
- `src/pages/allApps/index.tsx`
- `src/pages/ranking/index.tsx`
- `src/pages/searchList/index.tsx`
- `src/pages/customCategory/index.tsx`
- `src/pages/recommend/utils.ts`
- `src/pages/allApps/utils.ts`

**问题描述**：

部分页面早期通过 `generateEmptyCards()` 之类的假数据做占位，后续虽然逐步切向 `ApplicationCardSkeleton`，但“过滤 `empty-*` 假数据”的逻辑残留在多个页面的分页追加链路中。

**影响**：

- 加载态与业务数据耦合
- 列表拼接逻辑被污染
- 多页保留历史分支

**建议方向**：

彻底淘汰假数据占位，统一使用真实的 skeleton / loading 状态组件。

### P0-7 `ApplicationCard` 操作态与 loading 存在双重来源

**涉及模块**：`src/components/ApplicationCard/index.tsx`、`src/hooks/useApplicationCardModel.ts`

**问题描述**：

同一张卡片的操作类型和 loading 状态同时由：

- 上游 `getCardState(app)`
- 卡片内部 `resolvedOperateId`
- 本地 `buttonLoading`
- 以及 `isInstalling` 的同步 effect

共同决定。

**影响**：

- loading / 文案可能闪烁
- 逻辑归属不清
- 后续动作扩展（重试、暂停、打开）会更复杂

**建议方向**：

让 `useApplicationCardModel` 或独立 hook 直接产出完整 card view model，组件退化为展示层。

### P0-8 `ApplicationCard` 的 props 组装层在多个页面重复出现

**涉及模块**：

- `src/pages/recommend/index.tsx`
- `src/pages/allApps/index.tsx`
- `src/pages/ranking/index.tsx`
- `src/pages/searchList/index.tsx`
- `src/pages/customCategory/index.tsx`
- `src/pages/myApps/components/myApp/index.tsx`
- `src/pages/updateApp/index.tsx`

**问题描述**：

多个页面重复做：

- `const cardState = getCardState(item)`

并且已出现 `operateId={4}` 这类与枚举定义不一致的魔法数字漂移。

**影响**：

- 卡片契约过宽

- 参数漂移风险已出现

**建议方向**：

### P0-9 更新红点状态存在双份保存

**涉及模块**：`src/stores/updates.ts`、`src/stores/global.ts`、`src/hooks/useMenuBadges.ts`

**问题描述**：

当前菜单红点渲染主要已经依赖：

- `useUpdatesStore(state => state.updates.length)`

但 `updatesStore.checkUpdates()` 仍在反复写：

- `useGlobalStore.getState().getUpdateAppNum(...)`

而 `global.updateAppNum` 基本已接近遗留状态。

**已核对证据**：

- `useMenuBadges.ts` 明确读取 `updates.length`
- `updates.ts` 多次调用 `getUpdateAppNum(...)`
- `layout/sidebar/index.tsx` 还残留旧注释代码

**影响**：

- 双份事实来源
- 跨 store 写副作用
- 后续容易出现数量不同步

**建议方向**：

更新数量只保留一个事实来源，建议直接使用 `updates.length` 或统一 selector。

### P0-10 IPC 安装取消事件存在真实契约漂移

**涉及模块**：

- `src-tauri/src/services/install/progress_emitter.rs`
- `src-tauri/src/services/install/installer.rs`
- `src/types/api/invoke.d.ts`
- `src/hooks/useGlobalInstallProgress.ts`

**问题描述**：

Rust 侧真实会发出：

- `eventType: "cancelled"`

但 TS 类型 `InstallEventType` 中并未声明 `cancelled`，前端监听也未显式处理这一分支，导致取消事件会落入默认分支。

**影响**：

- 取消语义可能被按失败语义处理
- 契约已经发生真实偏差

**建议方向**：

统一安装事件类型定义，补齐 `cancelled` 判别分支。

---

### P0-11 Tauri `invoke` 入口没有完全收敛，存在绕过统一封装的调用

**涉及模块**：

- `src/apis/invoke/index.ts`
- `src/services/installService.ts`
- `src/components/speedTool/index.tsx`

**问题描述**：

虽然主入口是 `src/apis/invoke/index.ts`，但实际还有：

- `installService.ts` 直接再次 `invoke('install_app')`
- `SpeedTool` 组件内直接 `invoke('get_network_speed')`

**已核对证据**：

- `speedTool/index.tsx` 明确直接调用 `invoke<NetworkSpeed>('get_network_speed')`
- `installService.ts` 存在 `executeInstall()`，内部直接 `invoke('install_app')`

**影响**：

- IPC 契约分叉
- 错误处理策略重复
- 组件层直接接触 Tauri API

**建议方向**：

所有 IPC 调用统一经过单一 Tauri client 层，组件不再直接 `invoke`。

---

### P0-12 Rust `ll-cli` 适配层只有命令工厂，没有执行器 / 解析器 / 错误模型

**涉及模块**：`src-tauri/src/services/**/*.rs`、`src-tauri/src/utils/**/*`

**问题描述**：

仓库已经有 `ll_cli_command()`，但多个 service 仍在重复做：

- `.output()`
- `status.success()` 判断
- `stdout/stderr` 提取
- JSON 解析
- 错误字符串拼装

并且大量 `async fn` 内部直接执行阻塞式 `std::process::Command::output()`。

**已核对证据**：

在以下文件可看到大量重复：

- `src-tauri/src/services/prune.rs`
- `src-tauri/src/services/install/operations.rs`
- `src-tauri/src/services/linglong.rs`
- `src-tauri/src/services/linglong_env.rs`
- `src-tauri/src/services/process.rs`
- `src-tauri/src/services/install/installer.rs`

并存在：

- 多处 `.output()`
- `installer.rs` 中 `std::thread::sleep(...)`

**影响**：

- 错误处理不一致
- 缺少统一 timeout
- 解析逻辑分散
- 表面 async，内部 blocking

**建议方向**：

逐步建立统一的 `ll-cli adapter`：

- 执行器
- parser
- models
- 错误类型
- timeout 策略

5. 除安装流程外几乎没有统一 timeout 策略
6. 大量 `async fn` 实际执行阻塞式进程调用
7. `installer.rs` 中 `stderr` drain 存在缺口风险
8. `operations.rs`、`linglong_env.rs` 职责堆积明显
9. `#[allow(dead_code)]` 使用范围偏宽
