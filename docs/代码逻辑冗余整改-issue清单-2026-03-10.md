## ISSUE-01：统一启动初始化入口，消除重复加载与旧快照恢复问题

### 背景

当前启动阶段“加载已安装应用、补全详情、检查更新、恢复安装任务”存在多个入口同时触发，导致：

- 启动请求重复
- 状态覆盖顺序不清
- `recoverInstallTask(installedApps)` 存在读取旧 store 快照风险

### 改造目标

建立单一启动编排入口，让启动期核心副作用只由一个上层统一调度。

### 涉及文件

- `src/hooks/launch.ts`
- `src/layout/index.tsx`
- `src/stores/installedApps.ts`
- `src/stores/updates.ts`

### 建议改造内容

- 明确 `useLaunch` 是否作为唯一启动编排入口
- 收掉 `AppLayout` 中重复的初始化触发逻辑
- 收敛 `fetchInstalledApps()` 与 `updateAppDetails()` 的职责
- 修复恢复安装任务时读取旧 `installedApps` 快照的问题

### 验收标准

- 启动期每类核心请求只保留一个入口
- 不再重复调用 `updateAppDetails()`
- 安装任务恢复使用最新 store 数据
- 启动流程日志与执行顺序更清晰

### 风险说明

- 影响应用启动流程，需重点回归启动页、已安装应用列表、更新红点、安装恢复

### 推荐标签

- `refactor`
- `frontend`
- `state-management`
- `high-priority`

- `src/layout/launchPage/index.tsx`
- `src/components/LinglongEnvDialog/index.tsx`

- 清理重复的 `envReady / envChecked / error` 表达
- 收敛重试与自动安装后的检测链路
- 评估遗留字段如 `checked` 是否保留

### 验收标准

- 环境状态只有一套业务真相
- 用户点击一次重试只执行一轮必要检测
- 启动页与环境弹窗不再重复推进同一流程

### 风险说明

## ISSUE-11：为高频 IPC 返回补充运行时校验

### 改造目标

为高频、高影响 IPC 接口建立 runtime guard，尽早在边界层发现契约漂移。

### 优先范围

- `InstalledApp[]`
- `RunningApp[]`
- `LinglongEnvCheckResult`
- `InstallProgress`
- `NetworkSpeed`

### 涉及文件

- `src/apis/invoke/index.ts`
- `src/types/api/invoke.d.ts`
- 未来 validator/schema 位置

### 建议改造内容

- 引入 Zod schema 或等价 runtime validator
- 在统一 client / invoke helper 层做 parse
- 优先覆盖高频接口，避免一次性铺开过大

### 验收标准

- 高频 IPC 返回具备基础运行时校验
- 前端在边界层即可发现不兼容返回结构

### 风险说明

- 接入时可能暴露已有隐藏契约问题，属于好事，但会增加短期修复成本

### 推荐标签

- `enhancement`
- `contract`
- `ipc`
- `validation`

## ISSUE-15：统一 Rust 命令 timeout / blocking 策略

### 背景

当前多个 `async fn` 内部直接调用阻塞式 `std::process::Command::output()`，并存在 `std::thread::sleep(...)` 等 blocking 行为。

### 改造目标

统一 timeout 策略与异步执行模型，避免“表面 async，内部 blocking”。

### 涉及文件

- `src-tauri/src/services/install/installer.rs`
- `src-tauri/src/services/linglong_env.rs`
- `src-tauri/src/services/process.rs`
- `src-tauri/src/services/install/operations.rs`
- 其他直接 `.output()` 的 service

### 建议改造内容

- 对命令执行统一 timeout 策略
- 评估 `tokio::process::Command` 或 `spawn_blocking`
- 将 `std::thread::sleep()` 替换为更合适的 async 方式
- 补齐 `stderr` drain 处理，避免潜在阻塞

### 验收标准

- 主要命令执行不再直接阻塞 async 上下文
- timeout 行为统一可控
- 长任务执行更稳定

### 风险说明

- 风险较高，建议拆小步推进并重点回归安装、进程、环境检测能力

### 推荐标签

- `refactor`
- `rust`
- `performance`
- `async`
