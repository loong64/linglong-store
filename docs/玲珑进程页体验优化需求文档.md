# 玲珑进程页体验优化需求文档

| 项目 | 内容 |

## 2. 背景与现状

当前“玲珑进程”页本质上是一个前端表格页，用于展示正在运行的玲珑应用，并提供“进入容器”和“停止”两个操作。功能可用，但体验上更像调试页，不像桌面管理工具。

### 当前主要问题

- 页面视觉层级弱，主次信息不清晰。
- 操作入口集中在表格操作列，交互风格偏 Web 后台。
- 前端固定每 `1s` 轮询一次，刷新频率过高。
- 刷新方式是整表更新，容易出现闪烁和阻塞感。
- `rowKey` 使用索引，不稳定，存在不必要重渲染。
- Rust 端每次查询运行中应用时会对每个进程追加一次 `ll-cli info`，存在明显的 N+1 外部命令调用问题。
- 页面缺少更完整的状态表达，例如最近刷新时间、静默刷新、失败退避、行级操作中等。
- 当前尚未实现右键菜单，不符合桌面应用的常见操作习惯。

## 3. 需求目标

本次需求目标不是单纯美化表格，而是将“玲珑进程”页升级为更符合桌面应用使用习惯的进程管理页。

### 目标

- 提升页面流畅度，降低频繁刷新带来的卡顿和闪烁。
- 提升页面可读性，让用户更快识别当前运行中的应用。
- 将主要操作迁移到右键菜单。
- 保持现有能力不退化，继续支持停止进程、复制进入容器命令等能力。
- 优化前后端数据链路，降低进程页刷新成本。
- 保持现有项目架构风格，优先复用现有依赖和封装。

## 4. 非目标

- 不做完整的系统级进程监控面板。
- 不引入后台常驻事件监听服务。
- 不重构整个“我的应用”页面。
- 不扩展暂停、重启、资源占用监控等新能力。
- 不修改 `ll-cli` 本身行为。

## 5. 目标用户

- 普通桌面用户：查看正在运行的玲珑应用，并快速结束进程。
- 进阶用户：复制进入容器命令，在终端中进一步排查或调试。
- 高频用户：希望页面稳定、轻量，不因轮询造成明显卡顿。

## 6. 需求范围

本次需求范围包括以下内容：

- 进程页 UI 结构优化
- 列表刷新策略优化
- 行级右键菜单
- 行级操作状态与消息反馈
- 前端逻辑抽象与复用
- Rust 进程查询逻辑优化
- 文档、变更记录与自测补充

## 7. 右键菜单库选型

### 7.1 选型结论

本次右键菜单采用 `@tauri-apps/api/menu` 提供的原生菜单能力实现，不使用 Web 浮层菜单。

### 7.2 选型理由

- `@tauri-apps/api/menu` 调起的是系统原生上下文菜单，更符合桌面应用的交互习惯。
- 原生菜单弹出更直接，没有 Web 层动画拖沓感，符合“瞬时弹出”的产品要求。
- 菜单定位由 Tauri 原生能力处理，更接近桌面端右键体验。
- 当前需求以效率和桌面原生感为优先，原生菜单比 Web 组件菜单更合适。
- 不需要引入新的第三方右键菜单库。

### 7.3 不采用自定义右键菜单的原因

- 自定义浮层需要自行处理定位、边界、关闭行为和键盘交互，维护成本更高。
- 当前项目已有成熟 UI 库，不应重复造轮子。
- 当前需求优先桌面原生体验，直接使用 Tauri 原生菜单更合适，不需要额外引入 `react-contexify` 等第三方库。

### 7.4 参考资料

- Tauri Menu API：<https://v2.tauri.app/reference/javascript/api/namespacemenu/>
- Tauri Window Menu / Context Menu：<https://v2.tauri.app/learn/window-menu/>

## 8. 页面最终效果要求

优化后的页面应具备以下形态：

- 页面顶部有轻量工具栏，展示运行中数量、最近刷新时间、刷新按钮、自动刷新状态。
- 主体仍以列表或表格形式呈现，但视觉更轻，重点突出应用名和运行状态。
- 行右键可弹出上下文菜单，作为主要操作入口。
- 行 hover 时有明确反馈，右键打开菜单时有当前行选中态。
- 静默刷新时页面不应整表闪烁，也不应出现整页 loading。
- 正在停止中的进程仅该行显示操作中状态。
- 没有运行中进程时有明确空状态。
- 请求失败时有轻量错误提示，并支持手动刷新恢复。

## 9. 功能需求

### 9.1 页面结构

页面由三部分组成：

1. 顶部状态工具栏
2. 进程列表主体
3. 右键上下文菜单

### 9.2 顶部状态工具栏

顶部状态工具栏至少包含以下内容：

- 当前运行中进程数
- 最近刷新时间
- 手动刷新按钮
- 自动刷新状态展示
- 可选的自动刷新开关

### 9.3 进程列表主体

列表主体至少展示以下字段：

- 应用名称
- 版本号
- 架构
- 渠道
- 来源
- PID
- 容器 ID

说明：

- 应用名称作为主信息展示。
- `PID` 和 `容器 ID` 使用等宽字体。
- `架构`、`渠道`、`来源` 可以使用 `Tag`、弱化文本或徽标形式展示。
- 可保留一个轻量“更多”入口，作为右键菜单的显式兜底入口，但不再把操作列作为主交互。

### 9.4 右键菜单内容

右键菜单至少包含以下能力：

- 停止进程
- 复制进入容器命令
- 复制应用 ID
- 复制 PID
- 复制容器 ID
- 刷新进程列表

说明：

- “停止进程”使用危险态样式。
- 正在停止中的进程，相关菜单项应禁用。
- 菜单项文案应简短直接，不使用技术感过强的内部术语。

### 9.5 刷新策略

刷新策略必须从“固定高频轮询”改为“条件感知型刷新”。

#### 具体要求

- 仅当当前 tab 为“玲珑进程”时允许自动刷新。
- 仅当页面处于可见状态时允许自动刷新。
- 默认自动刷新间隔为 `3s`。
- 首次进入页面时立即拉取一次数据。
- 用户执行“停止进程”后，操作完成后立即补一次刷新。
- 用户点击“手动刷新”时立即触发一次刷新。
- 若上一轮请求未完成，下一轮自动刷新必须跳过，不允许并发堆积。
- 若连续刷新失败，应进入退避机制，逐步延长下一次自动刷新时间。
- 页面从不可见切换为可见时，应立即补一次刷新。
- 静默刷新不应触发整页 loading。

#### 退避建议

- 第 1 次失败：`3s`
- 第 2 次失败：`6s`
- 第 3 次及以后失败：`10s` 封顶

### 9.6 进程操作

#### 停止进程

- 触发“停止进程”后，仅锁定当前行。
- 停止成功后给出成功提示，并立即刷新列表。
- 停止失败后给出失败提示，并恢复该行可操作状态。

#### 复制进入容器命令

- 命令格式保持当前产品决策：`ll-cli enter <appId>`
- 点击后复制到剪贴板。
- 成功时提示“命令已复制到剪贴板，请粘贴到终端中执行”。
- 失败时提示明确错误信息。

#### 复制标识信息

- 支持复制应用 ID。
- 支持复制 PID。
- 支持复制容器 ID。
- 成功时统一提示“已复制”。

### 9.7 页面状态

页面需明确区分以下状态：

- 初次加载
- 静默刷新
- 无数据空状态
- 行级操作中
- 刷新失败但保留旧数据
- 全量失败且无数据

状态要求如下：

- 初次加载时可使用轻量 loading 或骨架屏。
- 已有数据时，如果刷新失败，不清空列表，只提示错误。
- 无运行中进程时显示空状态组件。
- 行级操作不应触发整页 loading。

## 10. UI / UX 设计要求

### 10.1 视觉方向

- 保持与现有项目主题一致。
- 页面不再呈现为“普通后台表格”。
- 顶部工具栏具备卡片化或轻量分区感。
- 应用名列作为主要视觉焦点。
- 技术字段弱化但保持可读性。
- hover、选中、行级操作中等状态具备明确视觉反馈。

### 10.2 右键菜单交互要求

- 行右键时打开右键菜单。
- 菜单显示位置跟随鼠标触发点或当前鼠标位置。
- 菜单关闭行为遵循系统原生菜单逻辑。
- 切换行右键时，菜单内容跟随当前行变化。
- 正在停止中的行，其危险操作菜单项处于禁用或 loading 状态。

## 11. 技术方案要求

### 11.1 前端结构调整

前端不应继续把所有逻辑堆在 `linglongProcess/index.tsx` 内。建议拆分如下：

- `useLinglongProcesses`
  - 管理进程列表拉取
  - 管理智能刷新
  - 管理失败退避
  - 管理行级操作状态
- `processActions`
  - 统一定义右键菜单和更多菜单使用的动作配置
- `ProcessTable` 或同类组件
  - 聚焦表格列和展示逻辑

### 11.2 右键菜单实现要求

右键菜单采用 `@tauri-apps/api/menu` 的原生菜单方案实现，不自行编写底层浮层。

实现要求如下：

- 使用 `Menu`、`MenuItem` 等 Tauri Menu API 构建上下文菜单。
- 通过 `menu.popup()` 在右键触发时弹出原生菜单。
- 行级 record 与菜单动作的映射必须统一管理。
- 菜单项点击后应回调到前端统一动作分发逻辑，避免分散处理。
- 右键菜单默认无额外前端动画，保持系统原生瞬时弹出效果。

### 11.3 前端数据与渲染要求

- 使用稳定 `rowKey`，禁止继续使用数组索引。
- 行级 loading 与整页 loading 分离。
- 刷新中的状态展示应尽量轻量。
- 与现有 `invoke` 层保持兼容，不新增零散的系统调用入口。

### 11.4 后端优化要求

后端重点优化 `src-tauri/src/services/process.rs`。

要求如下：

- 保持现有 IPC 接口兼容，尽量不扩大调用面。
- 保持错误输出可诊断，避免静默吞错。
- 若优化过程中新增或调整字段，必须同步更新 TS 类型和前端调用层。

### 11.5 兼容性要求

- 不破坏现有 `getRunningLinglongApps`、`killLinglongApp` 的调用语义。
- 不改变“进入容器”通过复制命令到剪贴板的现有产品决策。
- 需与 `src/main.tsx` 中的全局右键拦截逻辑兼容。
- 继续保持 Linux-only 运行前提，不引入跨平台包袱。

- 梳理进程页现有数据和交互逻辑
- 抽离统一 hook 管理刷新与操作
- 实现顶部状态工具栏
- 改造进程表格样式与列定义
- 增加稳定 `rowKey`
- 接入 Tauri 原生右键菜单
- 实现更多菜单入口并与右键菜单复用动作
- 增加空状态、静默刷新、行级 loading
- 处理可见性切换、tab 切换、请求并发控制
- 完成提示文案统一

### 12.2 后端开发任务

- 审视 `ll-cli ps` 输出结构与当前解析逻辑
- 优化 `get_running_linglong_apps()` 数据获取路径

- 补充必要日志，便于后续排查

### 12.3 类型与封装任务

- 补充或整理前端进程数据类型
- 如有必要，补充 `src/types/api/invoke.d.ts`
- 保持 `src/apis/invoke/index.ts` 的封装整洁
- 如新增 TS 辅助类型，保持命名与目录结构符合仓库规范

### 12.4 文档与记录任务

- 更新 `CHANGELOG.md`
- 将本次形成的约定补充到仓库说明中，至少记录：
  - 进程页采用智能刷新策略
  - 右键菜单使用 `@tauri-apps/api/menu`
  - 行级操作状态统一由单一 hook 管理

## 13. 验收标准

满足以下条件视为验收通过：

- 用户进入“玲珑进程”页时，可正常看到当前运行中的玲珑应用列表。
- 页面刷新时不出现明显整表闪烁。
- 页面不再以 `1s` 固定全量轮询方式工作。
- 页面不可见或非当前 tab 时，自动刷新暂停。
- 页面恢复可见后可自动补刷新。
- 右键任意行可弹出菜单，并显示在正确位置。
- 菜单项可正常执行停止进程、复制命令、复制标识等操作。
- 行级操作时仅当前行展示忙碌状态。
- 无运行中进程时展示明确空状态。
- Rust 端进程查询路径已完成性能优化。
- 文档、变更记录和相关约定已同步更新。

- 无运行中进程

## 16. 风险与注意事项

- `ll-cli ps` 输出格式若不稳定，Rust 解析需更谨慎。
- 若只做前端，不做后端查询优化，则页面流畅度提升有限。
- 若完全移除显式操作入口，可能影响右键菜单的可发现性，因此建议保留轻量“更多”入口。
- 若后续调整 IPC 字段，必须同步更新 TS 类型与调用层。

## 17. 需求结论

本次优化的核心不是把表格做得更花，而是把“玲珑进程”页从一个粗糙的调试列表，升级成一个低打扰、流畅、清晰、符合桌面工具习惯的进程管理页面。

右键菜单方案采用 `@tauri-apps/api/menu` 的原生上下文菜单能力实现，不新增第三方右键菜单库，也不自定义实现底层菜单，从而在满足需求的同时获得更强的桌面原生感和更干脆的交互反馈。
